# Word Fishing Leaderboard And Shared Teacher Admin Design

## Summary

`단어 낚시` 게임에도 `단어 짝 맞추기`와 같은 수준의 리더보드를 추가한다. 학생은 낚시 게임 완료 후 점수를 리더보드에 등록할 수 있고, 현재 학교와 학년 기준의 주간, 월간, 연간, 학교 전체 리더보드를 바로 확인할 수 있다.

교사 화면에서는 낚시 게임 리더보드도 이름 수정과 기록 삭제까지 지원해야 한다. 다만 교사 관리 화면이 복잡해지지 않도록, 활동마다 별도 관리 카드를 늘리는 대신 `활동 탭 + 기간 탭` 구조를 사용한다. 즉, 교사는 하나의 `리더보드 관리` 카드 안에서 `짝 맞추기`와 `단어 낚시`를 전환해 같은 방식으로 관리한다.

## Goals

- 단어 낚시 완료 화면에 리더보드 저장과 조회 흐름을 추가한다.
- 단어 낚시도 학교/학년 기준의 주간, 월간, 연간, 학교 전체 리더보드를 갖게 한다.
- 교사 화면에서 단어 낚시 리더보드 기록도 이름 수정과 기록 삭제를 할 수 있게 한다.
- 활동이 늘어나도 교사 화면 리더보드 관리 UI가 과도하게 길어지지 않게 유지한다.
- 짝 맞추기와 단어 낚시가 같은 리더보드 패턴을 공유하도록 정리해 이후 활동 확장을 쉽게 만든다.

## Non-Goals

- 짝 맞추기와 단어 낚시 점수를 하나의 통합 순위표로 합치지 않는다.
- 듣기/말하기 퀴즈 리더보드를 이번 범위에 추가하지 않는다.
- 학생 계정 기반 식별 체계나 학번 시스템을 새로 도입하지 않는다.
- 과거 주/월 기록 히스토리를 교사가 탐색하는 별도 관리 화면은 만들지 않는다.

## Recommended Approach

추천안은 `활동별 공통 리더보드 엔진 + 교사 화면 활동 탭` 구조다.

이 구조를 선택하는 이유는 다음과 같다.

- 학생 결과 화면은 짝 맞추기와 낚시 게임이 거의 같은 저장/조회 UX를 쓸 수 있다.
- 교사 화면은 `리더보드 관리` 카드를 하나만 유지하고, 그 안에서 활동만 전환하면 된다.
- Firestore 저장 경로와 권한 규칙을 공통화하면 이후 다른 게임을 붙일 때도 확장 비용이 낮다.
- 이름 수정 / 기록 삭제 로직을 활동별로 복제하지 않고 재사용할 수 있다.

## Alternatives Considered

### 1. 활동별 개별 리더보드 컬렉션과 개별 교사 카드

- 장점: 기존 `matchingLeaderboards`를 거의 그대로 복제하면 빠르게 붙일 수 있다.
- 단점: `teacher workspace`에 낚시 전용 관리 섹션이 추가되어 UI가 금방 복잡해진다.
- 단점: 이름 수정, 삭제, rules, fetch 로직이 활동마다 중복된다.

### 2. 활동별 공통 구조 + 교사 활동 탭

- 장점: 교사 화면 복잡도를 가장 잘 제어한다.
- 장점: 학생/교사 모두 같은 리더보드 패턴을 재사용할 수 있다.
- 장점: 데이터 모델과 권한 규칙을 `activityType` 기준으로 확장할 수 있다.
- 단점: 현재 매칭 전용 함수와 경로를 약간 리팩터링해야 한다.

### 3. 점수 통합 단일 리더보드

- 장점: 화면은 가장 단순해 보인다.
- 단점: 게임별 점수 성격이 달라 공정하지 않다.
- 단점: 교사와 학생 모두 어떤 게임 기록인지 헷갈리게 된다.

이번 문서에서는 `2. 활동별 공통 구조 + 교사 활동 탭`을 채택한다.

## User Experience

### 1. 학생 단어 낚시 완료 화면

단어 낚시 결과 카드 아래에 짝 맞추기 완료 화면과 같은 구조의 리더보드 섹션을 추가한다.

흐름은 아래와 같다.

1. 학생이 단어 낚시를 끝낸다.
2. 결과 화면에서 최종 점수, 정답 수, 오답 수, 놓친 수, 평균 반응 시간을 본다.
3. `리더보드에 점수를 등록하시겠습니까?` 질문이 보인다.
4. 학생이 동의하면 이름을 입력하고 저장한다.
5. 저장 후 현재 학교/학년 기준 리더보드가 같은 화면 아래에서 갱신되어 보인다.
6. 학생이 원하지 않으면 저장을 건너뛰고 리더보드만 볼 수도 있다.

### 2. 학생 리더보드 탭

학생이 보는 낚시 게임 리더보드는 짝 맞추기와 같은 탭 구조를 사용한다.

- `이 주의 영단어 왕`
- `이 달의 영단어 왕`
- `올해의 영단어 왕`
- `우리학교 단어 왕`

정렬 규칙은 기존과 같다.

- 점수 내림차순
- 동점이면 더 짧은 시간 우선
- 그래도 같으면 더 이른 기록 우선

학교 전체 탭에서는 학생 이름 옆에 학년을 함께 보여준다.

### 3. 교사 화면 리더보드 관리

교사 화면의 `리더보드 관리` 카드는 하나만 유지한다.

탭 구조는 아래 두 단계로 구성한다.

- 활동 탭
  - `짝 맞추기`
  - `단어 낚시`
- 기간 탭
  - `이 주의 영단어 왕`
  - `이 달의 영단어 왕`
  - `올해의 영단어 왕`
  - `우리학교 단어 왕`

교사는 현재 선택된 활동과 현재 선택된 기간의 목록만 본다. 항목 UI는 지금과 동일하게 유지한다.

- 순위
- 학생 이름
- 점수
- 시간
- `이름 수정`
- `기록 삭제`

즉, 교사 입장에서는 관리 카드 수가 늘어나지 않고, 현재 보고 있는 활동만 바꿔 가며 편집하게 된다.

### 4. 교사 이름 수정 흐름

1. 교사가 활동 탭에서 `단어 낚시`를 선택한다.
2. 기간 탭을 선택한다.
3. 특정 학생의 `이름 수정`을 누른다.
4. 새 이름을 입력하고 확인한다.
5. 시스템은 현재 학교, 현재 학년 기준의 낚시 리더보드 기간들에 대해 같은 학생 기록을 모두 찾아 새 이름으로 옮긴다.
6. 새 이름 기록이 이미 있으면 점수 우선 규칙으로 병합한다.
7. 완료 후 현재 활동의 리더보드를 다시 불러온다.

이 흐름은 짝 맞추기와 동일하며, 단지 대상 활동만 다르다.

### 5. 교사 삭제 흐름

1. 교사가 활동 탭에서 `단어 낚시`를 선택한다.
2. 특정 학생의 `기록 삭제`를 누른다.
3. 확인창에서 다시 묻는다.
4. 확인하면 현재 학교, 현재 학년 기준의 낚시 리더보드 기간들에서 해당 학생 기록을 삭제한다.
5. 완료 후 현재 활동의 리더보드를 다시 불러온다.

## Data Model

현재 구조는 `matchingLeaderboards/{scopeKey}/entries/{studentNameNormalized}`를 사용한다. 이번 변경에서는 활동 구분을 위해 리더보드 저장 구조를 `activityType` 기준으로 일반화한다.

권장 구조:

`activityLeaderboards/{activityType}__{scopeKey}/entries/{studentNameNormalized}`

예시:

- `activityLeaderboards/matching__schoolId__3__week__2026-W14/entries/김민준`
- `activityLeaderboards/fishing__schoolId__3__week__2026-W14/entries/김민준`
- `activityLeaderboards/fishing__schoolId__all__school_all__all-time/entries/김민준`

### Entry fields

```json
{
  "activityType": "fishing",
  "scopeKey": "schoolId__3__week__2026-W14",
  "schoolId": "schoolId",
  "schoolName": "도촌초등학교",
  "grade": "3",
  "studentName": "김민준",
  "studentNameNormalized": "김민준",
  "periodType": "week",
  "periodKey": "2026-W14",
  "score": 1320,
  "elapsedSeconds": 38,
  "correctCount": 8,
  "wrongCount": 1,
  "missCount": 1,
  "solvedPairs": null,
  "updatedAt": "serverTimestamp",
  "createdAt": "serverTimestamp"
}
```

### Activity-specific metrics

짝 맞추기와 낚시 게임은 공통 필드와 활동별 필드를 함께 가진다.

- 공통 필드:
  - `activityType`
  - `scopeKey`
  - `schoolId`
  - `schoolName`
  - `grade`
  - `studentName`
  - `studentNameNormalized`
  - `periodType`
  - `periodKey`
  - `score`
  - `elapsedSeconds`
  - `createdAt`
  - `updatedAt`
- 짝 맞추기 전용 보조 필드:
  - `solvedPairs`
- 낚시 게임 전용 보조 필드:
  - `correctCount`
  - `wrongCount`
  - `missCount`

교사 목록과 학생 리더보드 탭에서는 공통 필드만으로 표시할 수 있게 유지한다. 활동별 보조 필드는 상세 비교나 미래 확장을 위한 보존 데이터로 둔다.

## Save Rules

### 공통 규칙

- 학교, 학교 이름, 학년, 학생 이름이 없으면 저장하지 않는다.
- 기간 키는 한국 시간 기준으로 계산한다.
- 주간, 월간, 연간, 학교 전체는 각각 독립적으로 처리한다.
- 기존 기록보다 점수가 높을 때만 갱신한다.
- 점수가 같으면 더 짧은 시간이 우선이다.
- 그것도 같으면 기존 기록을 유지한다.

### 낚시 게임 저장 입력

낚시 게임 저장 시 아래 값을 사용한다.

- `score`
- `elapsedSeconds`
- `correctCount`
- `wrongCount`
- `missCount`

학생이 단어 낚시를 완료했을 때 이 값으로 현재 기간들에 대해 upsert 한다.

## Teacher Admin Rules

교사 관리 기능은 현재 구조와 동일한 정책을 유지하되, 대상 활동을 선택해서 적용한다.

### 이름 수정

- 현재 학교 기준
- 현재 학년 기준
- 선택한 활동 기준
- 현재 주/월/연/학교전체 기간 기준

각 기간에 대해 아래 규칙을 적용한다.

1. 기존 이름 기록이 없으면 그 기간은 건너뛴다.
2. 새 이름 기록이 없으면 이동한다.
3. 새 이름 기록이 있으면 더 좋은 기록을 남기고 병합한다.
4. 병합 기준은 기존 리더보드와 동일하다.

### 삭제

- 현재 학교 기준
- 현재 학년 기준
- 선택한 활동 기준
- 현재 주/월/연/학교전체 기간 기준

각 기간의 해당 학생 문서를 삭제한다.

## UI Simplification Rules

교사 화면이 복잡해지지 않게 하기 위한 원칙은 아래와 같다.

- `리더보드 관리` 카드는 하나만 유지한다.
- 활동별로 카드나 섹션을 추가하지 않는다.
- 편집 입력창은 현재처럼 한 항목에서만 열리게 유지한다.
- 활동 탭과 기간 탭을 동시에 보여주되, 목록은 항상 하나만 보이게 한다.
- 상태 메시지와 에러 메시지는 카드 상단 공통 위치를 그대로 쓴다.

즉, 화면의 복잡도는 `카드 수`가 아니라 `탭 수`만 조금 늘어나는 수준으로 제한한다.

## Component Impact

### 학생 화면

- `WordFishingGame.jsx`
  - 완료 화면에 리더보드 저장/조회 UI 추가
- 공통 리더보드 요약 UI는 가능하면 `WordMatchingGame.jsx`와 공유 컴포넌트로 분리한다.

### 교사 화면

- `TeacherWorkspace.jsx`
  - 활동 탭 추가
  - 현재 탭의 활동에 따라 목록을 전환
  - 편집 / 삭제 액션은 공통 핸들러에 `activityType`을 넘겨 동작

### Hooks

- `useVocabularyLibrary.js`
  - 현재 teacher leaderboard state를 활동 기준으로 확장
  - fetch, rename, delete 함수에 `activityType` 인자 추가
  - 기본 활동 탭은 `matching`

### Firebase layer

- `firebase.js`
  - 매칭 전용 함수들을 공통 리더보드 함수로 일반화
  - 낚시용 fetch/save 함수 추가 또는 공통 함수 래핑
  - 교사 rename/delete도 활동 기준으로 동작하도록 확장

### Rules

- `firestore.rules`
  - 기존 매칭 리더보드 규칙을 공통 리더보드 경로에 맞춰 조정
  - 학생 저장 허용 범위는 `activityType`과 허용 필드 조합으로 제한
  - 교사 수정/삭제는 현재 학교 소속 교사만 가능하게 유지

## Migration Strategy

기존 매칭 리더보드 데이터가 이미 운영 중이므로, 새 구조로 갈 때는 아래 두 가지 중 하나를 선택해야 한다.

### 추천 마이그레이션

- 읽기 함수는 우선 새 공통 경로를 먼저 보고, 없으면 기존 `matchingLeaderboards`를 fallback 으로 읽는다.
- 쓰기 함수는 새 공통 경로에만 저장한다.
- 교사 편집도 새 공통 경로를 우선 사용한다.
- 필요하면 별도 관리 스크립트로 기존 매칭 데이터를 새 경로로 옮긴다.

이 방식은 운영 중 위험이 가장 낮다.

## Error Handling

- 낚시 게임 리더보드 저장 실패 시 결과 화면은 유지하고 오류 메시지만 보여준다.
- 일부 기간 저장 실패 시 기존 매칭과 같은 패턴으로 부분 성공 메시지를 보여준다.
- 교사 편집/삭제 중 일부 기간만 실패하면 성공/유지/실패 요약을 함께 보여준다.
- 활동 탭 전환 시 데이터가 없으면 `아직 관리할 리더보드 기록이 없습니다.`를 표시한다.

## Verification

1. 단어 낚시 완료 화면에 리더보드 저장 질문이 표시된다.
2. 학생이 이름을 입력해 저장하면 낚시 리더보드 문서가 생성된다.
3. 같은 학생의 더 낮은 낚시 점수는 기존 기록을 덮어쓰지 않는다.
4. 더 높은 낚시 점수는 해당 기간 기록을 갱신한다.
5. 교사 화면 리더보드 관리 카드에 `짝 맞추기 / 단어 낚시` 활동 탭이 보인다.
6. 교사가 낚시 리더보드에서 이름 수정과 기록 삭제를 할 수 있다.
7. 활동 탭을 바꿔도 교사 화면 레이아웃이 크게 늘어나지 않는다.
8. 짝 맞추기 리더보드 기존 동작이 유지된다.
9. `npm run build`가 통과한다.

## Open Decisions Resolved

- 낚시 게임도 짝 맞추기와 동일한 수준의 리더보드 저장/조회 흐름을 갖는다.
- 교사 화면에서도 낚시 게임 리더보드를 같은 수준으로 편집할 수 있어야 한다.
- 교사 화면은 활동별 별도 관리 섹션을 늘리지 않고 `활동 탭 + 기간 탭` 구조를 사용한다.
- 짝 맞추기와 낚시 점수는 섞지 않고 활동별로 분리한다.
