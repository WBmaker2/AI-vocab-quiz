# Teacher Leaderboard Admin Design

## Summary

교사 관리 화면에 `리더보드 관리` 섹션을 추가한다. 이 섹션은 현재 선택한 학년을 기준으로 같은 학교의 현재 주간, 월간, 연간 리더보드를 불러와 관리할 수 있게 한다.

교사는 리더보드 항목에 대해 두 가지 작업을 할 수 있다.

- 학생 이름 수정
- 리더보드 기록 삭제

이름 수정은 같은 학교와 같은 학년 안에서 현재 주간, 월간, 연간 리더보드에 걸려 있는 동일 학생 이름 기록을 한 번에 새 이름으로 바꾼다. 새 이름에 이미 기록이 있으면 더 높은 점수를 남기고 병합한다.

기록 삭제는 같은 학교와 같은 학년 안에서 현재 주간, 월간, 연간의 해당 학생 기록을 한 번에 삭제한다.

## Goals

- 교사가 장난으로 입력된 학생 이름을 삭제할 수 있게 한다.
- 교사가 오타로 입력된 학생 이름을 현재 기간 리더보드 전체에서 한 번에 수정할 수 있게 한다.
- 학생 화면은 계속 점수 등록만 가능하고, 리더보드 수정 권한은 교사에게만 유지한다.
- 현재 선택 학년 기준으로 리더보드 관리 범위를 분명하게 한다.

## Non-Goals

- 과거 주간/월간 리더보드까지 모두 수정하는 히스토리 관리 화면은 만들지 않는다.
- 학생 계정 시스템을 도입하지 않는다.
- 학교를 넘나드는 리더보드 관리 기능은 만들지 않는다.
- 듣기/말하기 퀴즈 점수를 리더보드 관리 범위에 포함하지 않는다.

## User Experience

### 1. 위치

- `TeacherWorkspace` 안에 `리더보드 관리` 섹션을 추가한다.
- 이 섹션은 현재 선택 학년을 기준으로 동작한다.
- 단원 입력과는 무관하지만, 학년 선택 상태에 따라 보여주는 데이터가 달라진다.

### 2. 표시 방식

- 교사 화면의 리더보드 관리 섹션은 탭형으로 구성한다.
  - `이 주의 영단어 왕`
  - `이 달의 영단어 왕`
  - `올해의 영단어 왕`
- 각 탭은 현재 학교 + 현재 선택 학년 범위의 상위 기록을 보여준다.
- 항목에는 아래 정보를 표시한다.
  - 순위
  - 학생 이름
  - 점수
  - 걸린 시간

### 3. 수정 흐름

1. 교사가 리더보드 항목의 `이름 수정` 버튼을 누른다.
2. 이름 수정용 입력 UI가 열린다.
3. 새 이름을 입력하고 확인을 누른다.
4. 확인창에서 정말 수정할지 다시 묻는다.
5. 확인하면 현재 학교 + 현재 학년의 현재 주간/월간/연간 기록에서 같은 학생 이름을 모두 찾아 새 이름으로 이동한다.
6. 이동 대상 이름이 이미 존재하면 더 높은 점수를 유지하고 기록을 병합한다.
7. 완료 후 현재 탭과 다른 탭도 포함해 리더보드를 다시 불러온다.

### 4. 삭제 흐름

1. 교사가 리더보드 항목의 `기록 삭제` 버튼을 누른다.
2. 확인창에서 정말 삭제할지 다시 묻는다.
3. 확인하면 현재 학교 + 현재 학년의 현재 주간/월간/연간 기록에서 해당 학생 이름을 모두 삭제한다.
4. 삭제 후 리더보드를 다시 불러온다.

## Scope Rules

리더보드 관리 범위는 아래 조건을 모두 만족하는 현재 기록으로 제한한다.

- 현재 로그인한 교사의 `schoolId`
- 현재 선택한 `grade`
- 현재 시점 기준의 `week`, `month`, `year` period key

즉, 같은 학교라도 다른 학년 기록은 건드리지 않는다.

## Data Model Impact

기존 리더보드 저장 구조를 그대로 사용한다.

`matchingLeaderboards/{scopeKey}/entries/{studentNameNormalized}`

### Existing fields reused

- `scopeKey`
- `schoolId`
- `schoolName`
- `grade`
- `studentName`
- `studentNameNormalized`
- `periodType`
- `periodKey`
- `score`
- `elapsedSeconds`
- `solvedPairs`
- `createdAt`
- `updatedAt`

## Rename Rules

이름 수정은 현재 주간/월간/연간 3개 기간을 각각 독립적으로 처리한다.

### 기본 규칙

1. 기존 학생 이름 기록이 없으면 그 기간은 건너뛴다.
2. 새 이름 기록이 없으면:
   - 기존 문서를 새 이름 key로 복사
   - `studentName`, `studentNameNormalized`, `updatedAt` 갱신
   - 기존 문서는 삭제
3. 새 이름 기록이 이미 있으면:
   - 두 기록의 점수를 비교
   - 더 높은 점수를 최종 기록으로 남긴다
   - 점수가 같으면 더 짧은 시간 기록을 우선한다
   - 최종 기록을 새 이름 문서에 저장
   - 기존 문서는 삭제

### 병합 우선순위

1. `score`가 높은 기록
2. 동점이면 `elapsedSeconds`가 더 짧은 기록
3. 그 외 필드는 최종으로 남은 기록 기준으로 유지

## Delete Rules

삭제는 현재 주간/월간/연간 각각에 대해 같은 학생 이름 key의 문서를 찾아 삭제한다.

결과는 아래 세 범주로 요약할 수 있다.

- 삭제됨
- 해당 기간에 기록 없음
- 오류

## Permissions

- 학생은 계속 리더보드 등록만 가능하다.
- 교사 리더보드 관리 기능은 로그인한 교사 화면에서만 노출한다.
- Firestore rules는 교사의 학교 리더보드 관리 권한을 지원해야 한다.
- 이상적인 규칙은 `teacher` 문서의 `schoolId`와 수정 대상 리더보드의 `schoolId`가 같을 때만 수정/삭제를 허용하는 방식이다.

## Error Handling

- 현재 학교 정보가 없으면 리더보드 관리 섹션 대신 안내 메시지를 표시한다.
- 현재 학년이 없으면 리더보드를 불러오지 않는다.
- 수정 이름이 비어 있으면 저장하지 않는다.
- 수정 또는 삭제 실패 시 사용자용 오류 메시지를 보여준다.
- 현재 기록이 없으면 `아직 관리할 리더보드 기록이 없습니다.`를 표시한다.

## UI Notes

- 탭형 구성은 학생 완료 화면 리더보드와 비슷한 시각적 언어를 사용해 일관성을 유지한다.
- `이름 수정`은 작은 보조 버튼, `기록 삭제`는 경고 색상을 쓰는 보조 버튼으로 처리한다.
- 삭제 버튼은 과도하게 눈에 띄지 않되, 위험 동작임을 분명히 표시한다.
- 모바일에서도 탭과 항목 버튼이 눌리기 쉽도록 세로 정렬을 허용한다.

## Firebase / Rule Expectations

현재 학생 점수 등록은 브라우저 직접 쓰기 구조라 완전한 무결성은 어렵다. 반면 교사 리더보드 관리 기능은 로그인된 교사만 사용하므로, 쓰기/삭제 규칙은 더 강하게 제한할 수 있다.

구현 시 필요한 방향:

- 교사 로그인 필수
- 수정/삭제 대상 리더보드의 `schoolId`가 현재 교사 `schoolId`와 같아야 함
- 수정은 현재 구조상 `studentNameNormalized`가 문서 key이므로 사실상 `delete + recreate` 또는 `copy + delete` 흐름이 필요

## Verification Plan

구현 후 아래를 검증한다.

1. 교사 화면에 `리더보드 관리` 섹션이 보인다.
2. 현재 선택 학년 기준으로 주간/월간/연간 탭이 보인다.
3. `이름 수정` 시 현재 주/월/연 기록이 한 번에 새 이름으로 바뀐다.
4. 새 이름에 기존 기록이 있을 때 더 높은 점수만 남기고 병합된다.
5. `기록 삭제` 시 현재 주/월/연 기록이 한 번에 삭제된다.
6. 수정/삭제 후 탭 데이터가 즉시 새로고침된다.
7. 다른 학교 리더보드에는 영향을 주지 않는다.
8. `npm run build`가 통과한다.

## Open Decisions Resolved

- 교사 리더보드 관리는 별도 페이지가 아니라 교사 관리 화면 내부에 둔다.
- 관리 범위는 현재 선택 학년으로 제한한다.
- 이름 수정은 현재 주/월/연 기록을 한 번에 처리한다.
- 기록 삭제도 현재 주/월/연 기록을 한 번에 처리한다.
